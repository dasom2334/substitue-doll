# /review — 작업 후 자동 리뷰 (CLAUDE.md §10)

방금 변경(현재 브랜치/PR)에 대해 §10 자동 리뷰를 수행한다. 이 스킬이 **오케스트레이터**이고, 게시는 이 스킬이 한다(리뷰어 서브에이전트는 결과 텍스트만 반환).

## 절차
1. **현재 PR 번호 확인**: `gh pr view --json number -q .number`.
   - 열린 PR이 없으면: 리뷰는 진행하되 게시 대신 전문을 사용자에게 출력하고 "게시할 PR 필요"를 알린다(§10). 리뷰를 버리지 않는다.
2. **리뷰 대상 파악**: `gh pr diff {번호} --name-only`로 변경 파일 확인.
   - 코드/설정 포함 → 정식 리뷰. 문서(.md) 전용 → 가볍게 보되 **게시는 동일하게**(모든 리뷰 기록).
3. **`reviewer-correctness` 호출**(블로킹 게이트): 방금 변경을 리뷰.
   - **REQUEST_CHANGES** → 거기서 멈춘다. 각 문제에 **구체적 수정안**을 제시하고 사용자에게 보고. **사용자 명시 지시 전 코드 수정 금지.** (단 코멘트 게시는 5로 진행 — 모든 리뷰는 기록)
   - **APPROVE** → 4로.
4. **`reviewer-design` 호출**(권고): **정확성 결과는 넘기지 말고** 코드만 독립 판단(독립성 유지).
5. **PR 코멘트 게시**(필수, 생략 불가): 아래 §10 형식으로 `gh pr comment {번호} --body-file <파일>`. 실패 시 최대 2회 재시도 → `gh auth status` 확인 → 그래도 실패면 전문 출력하고 "수동 게시 필요" 알림.
6. **사용자에게 종합 보고.**

## 게시 형식 (한국어)
```
## 🔍 리뷰 결과 — {판정}
### 정확성·안전
{발견 사항. 없으면 "문제 없음". 각 항목 {파일}:{줄}}
### 설계·가독성 (권고)
{제안 사항. 각 항목 {파일}:{줄} + 왜 더 나은지}
### 💡 Python 팁
{이번 변경에서 짚을 만한 Python 팁(실무 패턴/관용구), 간단히}
### 최종 판정: {APPROVE | REQUEST_CHANGES}
```

## 규칙
- 순서: **정확성(게이트) → 통과 시 설계(권고)**. 코드 수정은 사용자 지시에서만.
- 모든 리뷰는 판정과 무관하게 게시(생략 불가).
- 리뷰어가 로드돼 있지 않으면(`reviewer-correctness`/`reviewer-design` 사용 불가) 사용자에게 알리고, 세션 재로드 후 실행.
