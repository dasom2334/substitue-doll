# CLAUDE.md — 매 세션 따를 실행 규칙

> 이 파일은 **실행 규칙만** 담는다. 로드맵·상세 배경·미정 사항은 PLAN.md에 있다.
> 규칙(여기)과 상세 배경(PLAN.md)은 분리한다. 규칙에 꼭 필요한 근거만 한 줄 각주로 붙이되, 그 근거를 빌미로 규칙을 느슨하게 풀지 마라.

## 0. 이 프로젝트가 무엇인가 (잊지 말 것)
- 사용자를 능동적으로 **이끄는 AI 에이전트**다. 단순 말투 재현 도구가 아니다.
- **하네스(에이전트의 능동적 인도 행동)가 이 프로젝트의 심장이고, 말투 재현은 그 그릇일 뿐이다.**
  - **(시즌2 고정 내용)** 하네스가 하는 일: 고찰, 너무 나쁜 방향으로 가는 생각 잡아주기, 객관적으로 보게 하기, 건강한 미래로 인도 — 그리고 **힘든 과정을 건강하게 이겨내도록 함께하는 동반자**로 동작. 시즌2에선 도구의 목적을 **"이별 마음 정리를 돕는"** 것으로 고정한다. (관계 분석은 범위 제외.)
  - 말투만 재현하고 하네스가 빠지면 사용자가 원하는 결과물이 **아니다.** 하네스를 "나중에 칠하는 살"로 미루지 마라. 어떤 기능을 설계하든 하네스를 중심에 두고 판단하라.
- 에이전트는 **답변을 "생성"만** 한다. 자동 발송/실행(예: 메신저 API 직접 전송)은 하지 않는다.
- **과몰입 방지(클론 규약):** 시즌2 클론은 **처음부터** "나는 그 사람의 클론"이라 고지하고, 대화 내내 클론임을 분명히 한다. 사용자가 클론을 **정말 그 사람처럼 여겨 빠져들지 않도록** 흐름 자체에 방지를 둔다 — 클론임을 지속적으로 환기하고, 진짜 그 사람인 척하는 방향으로 흐르지 않게 한다.

## 1. 작업 완료 기준 (커밋 게이트 — 불변)
- 커밋 전에 **ruff(lint) + mypy(typecheck) + pytest가 전부 통과**해야 한다. 하나라도 실패하면 **커밋 금지.**
- 이 게이트는 **pre-commit 훅으로 강제**한다(스캐폴딩 0단계에서 설정).

## 2. 커밋 규칙
- **원칙: 커밋은 최대한 잘게 쪼개 하나의 역할만 하도록 한다.** 한 커밋엔 **하나의 논리 변경**만. 여러 역할이 섞이면 나눠서 커밋한다.
- 작업 하나 완성 + 테스트 통과 시 **즉시 커밋.**
- 메시지 형식: `feat/fix/chore(scope): 설명`.

## 3. 비밀키 / 설정
- API 키는 **`.env` + 환경변수로만.** `.env`는 `.gitignore`에 둔다.
- config 코드는 환경변수를 **읽기만** 한다. 키를 코드/리포에 하드코딩하지 않는다.

## 4. 데이터 경계 (정제 실행 규칙)
- 당사자가 둘 사이에서만 발화할 수 있는 외부 데이터나 사적 정보는 **제거**한다.
  - 예외: **직업·MBTI 같은 특정 정보는 보존** 가능.
- 둘 사이의 **사건·이벤트는 저장 가능.** "X를 했다"도 저장한다 — 거기 **묶인 감정을 저장하기 위해서**다. 사건과 감정은 함께 저장한다.
- 위 제거를 거치고 **남은** raw 원본 말투는 저장 가능.
- **정제 먼저.** 원본은 raw 그대로 영구 저장하지 않는다.
- 근거(주의): 이 경계가 막는 것은 "복제물 보유"가 아니라 **도덕적으로 민감한 부분 회피**다. 직업·MBTI 등 특정 정보를 **보존**하는 건 그게 있어야 **퀄리티가 높아지기** 때문 — 단, 이 근거를 빌미로 제거 규칙을 **느슨하게 풀지 마라.**

## 5. 입력 정책
- 입력은 **자유 형식**(평문이든 카톡 로그든 무엇이든). 시즌1·2 둘 다 동일.
- 카톡 로그가 없으면 설명한 대로만, 있으면 정제를 전부 수행한다.
- UI 유도는 **"정보를 넣어주세요" 정도로만 모호하게**(유명 AI 제품들처럼).
- **"데이터 인입"과 "유저가 클론과 대화하는 부분"은 서로 다른 것이다.**
  - "받는 즉시 정제한다"는 **데이터 인입**에 대한 규칙이지, 클론과의 실시간 대화에 대한 규칙이 아니다. 둘을 구분해 설계하라.
  - **런타임 대화 중** 사용자가 새로 흘리는 사적 정보도 **인입과 동일하게 즉시 정제 후 저장**한다(§4 규칙 적용). 정제 없이 raw로 저장하지 않는다.

## 6. 데이터 위생 (불변)
- 정제 구현 전(스텁 단계)에는 **합성/더미 데이터로만** 테스트한다.
- 실제 제3자가 포함된 로그는 **정제가 완성된 뒤에만** 투입한다.
- 원본은 **어떤 단계에서도** raw로 영구 저장하지 않는다.

## 7. 작업 위생
- 요청받지 않은 파일/기능은 건드리지 않는다.
- 작업은 **한 번에 하나.**

## 8. 아키텍처 — 엔진과 껍데기(인터페이스) 분리 (불변)
- **엔진(`core/engine`)에 UI/API/입출력 관련 코드를 절대 넣지 않는다.** 엔진은 입력을 받아 결과를 반환하는 **순수 함수**로만 유지한다.
- CLI/Streamlit/FastAPI 등 인터페이스는 엔진을 **호출하는 얇은 껍데기**로만 둔다. 로직을 껍데기에 품지 않는다.
- **엔진은 어떤 껍데기가 자기를 호출하는지 알아서는 안 된다.** (기존 "코어는 시즌을 모른다" 규칙의 연장 — 코어는 어떤 시즌인지도, 누가 부르는지도 몰라야 한다.)
- 여러 껍데기는 같은 엔진 함수를 호출만 하므로 공존 가능하다. 인터페이스 추가/교체 시 엔진은 **한 줄도 건드리지 않는다.**

## 9. Git 워크플로우 (불변)
- **`main` 브랜치에 직접 커밋/푸시하지 않는다.**
- 모든 작업은 feature 브랜치(`feat/...`, `fix/...`)에서 하고, 완료되면 **PR로 올린다.**
- **PR은 절대 스스로 머지하지 않는다. 머지는 사용자만 한다.**
- PR을 올린 뒤에는 사용자의 리뷰/머지를 기다린다. **머지를 가정하고 다음 작업을 진행하지 않는다.**

### PR 본문 포맷 (빌더가 PR 생성/갱신 시)
PR을 만들거나 본문을 갱신할 때 아래 형식을 따른다(한국어):
```
## 작업
{PLAN.md 작업 번호 + 무엇을 했는지 한 줄}
## 변경 내용
- {파일/모듈별 핵심 변경}
## 테스트
- {추가/통과한 테스트, ruff·mypy·pytest 결과}
## 체크리스트
- [ ] 정제 안 된 raw 데이터 저장 경로 없음
- [ ] core/engine 분리 유지
- [ ] 민감/대용량 파일 커밋 제외(.env, 카톡 원본, 벡터DB)
```
- PR 본문은 **빌더(작업자)가 작성/갱신**한다. **리뷰어는 본문을 건드리지 않고 코멘트만 단다.**

## 10. 작업 후 자동 리뷰 (불변)
코드 작업을 완료하고 커밋해 PR에 푸시한 **직후**, 사용자가 따로 요청하지 않아도 아래를 **자동으로** 수행한다. 리뷰는 로컬 세션 안에서 서브에이전트로 돈다.

1. **`reviewer-correctness`**(정확성·안전, 블로킹 게이트)를 호출해 방금의 변경을 리뷰한다.
2. 결과가 **REQUEST_CHANGES**면 → **거기서 멈춘다.** 각 문제에 대한 **구체적 수정안을 함께 제시**한 뒤 사용자에게 보고한다.
   - **사용자가 명시적으로 지시하기 전에는 코드를 수정하지 않는다.** (사용자가 제시안을 보고 "적용해 / 일부만 / 무시"를 판단한다.)
3. 결과가 **APPROVE**면 → **`reviewer-design`**(설계·가독성, 권고)을 호출한다. 이때 **정확성 리뷰 결과는 넘기지 말고 코드만 보게 한다**(독립성 유지).
4. 두 리뷰 결과를 **PR 코멘트로 게시**한다(아래 'PR 코멘트 게시 규칙').
5. 게시 후 사용자에게도 결과를 **종합 보고**한다.

→ 순서는 **"정확성(게이트) → 통과 시 설계(권고)"**이며, **코드 수정은 언제나 사용자의 명시적 지시에서만 시작**한다.

### PR 코멘트 게시 규칙 (두 리뷰어 공통, 필수)
- 판정과 무관하게(APPROVE든 REQUEST_CHANGES든) **항상 PR에 코멘트로 게시**한다.
- 현재 PR 번호는 `gh pr view --json number` 또는 `gh pr list`로 확인한다.
  - **열린 PR이 없으면** 게시는 건너뛰되, 리뷰 결과는 사용자에게 그대로 보고한다.
- 게시 명령: `gh pr comment {PR번호} --body "리뷰 내용"`
- 게시 형식(한국어):
  ```
  ## 🔍 리뷰 결과 — {판정}
  ### 정확성·안전
  {발견 사항. 없으면 "문제 없음". 각 항목 {파일}:{줄}}
  ### 설계·가독성 (권고)
  {제안 사항. 각 항목 {파일}:{줄} + 왜 더 나은지}
  ### 💡 Python 배우기
  {이번에 짚은 실무 패턴/관용구 설명}
  ### 최종 판정: {APPROVE | REQUEST_CHANGES}
  ```
- 게시 실패 시(**최대 2회 재시도**): `gh auth status`로 인증 확인 후 재시도. 그래도 실패하면 리뷰 내용을 그대로 출력하고 **"PR에 수동 게시 필요"**라고 알린다.

### 빌더 응답/수정 코멘트 (빌더가 리뷰에 응답/수정 완료 시)
리뷰어 지적에 빌더가 응답하거나 수정 완료를 알릴 때, 아래 형식으로 PR 코멘트를 단다(한국어):
```
## 🔧 수정 완료 — {커밋 해시}
- {지적 N}: {어떻게 고쳤는지}
- 적용 안 함: {무시한 지적 + 이유}
```
- 수정은 **사용자의 명시적 지시 이후에만** 시작한다(§10 흐름). 적용/미적용 판단의 근거를 남긴다.

### 리뷰 공통 규칙
- 모든 출력은 **한국어**. 모든 판단에 **근거**를 붙인다.
- 리뷰어는 **코드를 재작성하지 않는다**(개념 설명용 짧은 예시 스니펫은 허용).
- **무조건 칭찬 금지.** 단 잘한 부분은 짚어 학습을 강화한다.
- 같은 Python 실수가 반복되면, 아래 **"자주 하는 실수"** 섹션에 추가하도록 사용자에게 **제안**한다(직접 수정 말고 제안만).

## 11. 자주 하는 실수 (Python) — 학습 누적
> 리뷰에서 **반복되는** Python 실수가 확인되면, 사용자 동의 후 여기에 한 줄씩 누적한다. (리뷰어는 직접 수정하지 않고 제안만 한다.)
