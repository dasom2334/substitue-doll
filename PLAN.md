# PLAN.md — 로드맵

> 이 파일은 로드맵·배경·미정 사항을 담는다. 매 세션 따를 실행 규칙은 CLAUDE.md에 있다.
> **A(사용자 확정)만 사실이다. B(미정)는 추측으로 채우지 않는다 — 우선순위대로 사용자에게 물어 정한다.**

---

## 0. 배경 / 철학 (왜 이렇게 설계하는가 — 배경일 뿐, 실행 규칙 아님)

- 동기(근거가 되는 현상):
  - 사람들이 **AI에게 발화·대화를 조언받는** 케이스가 이미 많다.
  - **실제로 대화할 수 없는 대상에게 "보낼 수 없는 편지"를 쓰는 것이 회복에 도움이 된다**는 임상 케이스가 있다.
  - 실제로 **AI에게 정서적 도움을 받는** 케이스가 존재한다.
- 데이터 경계를 왜 그렇게 그었는지(윤리적 근거)와 직업·MBTI 보존 예외의 이유는, 규칙과 함께 **CLAUDE.md §4 근거 각주**에 둔다.

## 1. 모드 구조

### 모드1 — 감쓰통 / 공감 (MVP 1차 목표)
- 친구가 나를 감정쓰레기통으로 쓸 때 **대리로 답해주는** 에이전트.
- 용도: 감쓰통, 친밀도 강화용(썸/연애/친구), 공감용.
- 입력: **어떤 포맷이든** 내 과거 대화 기록으로 말투를 학습시킨다(자유 형식 인풋 — 1단계 참조).
- 동작: 내가 **직접 발화해야 하거나 대답하기 곤란한 상황**에서, 추천 대화(답변 초안)를 **생성·어드바이징**해 일종의 **"대리 출동"**을 한다. 답변을 **"생성"만** 하고 직접 발송 API는 쏘지 않는다.

### 모드2 — 이별·상실 정리 대리자 (프로덕션 지향, 하네스의 본진)
- 형태: "이별·상실을 돕는 대리자(전남친 버블류)." 떠나보낸 **대상(전 연인·반려동물·고인 등)**의 대리자와 대화하며 **이별·상실을 정리**한다.
- 대리자 캐릭터 규약:
  - **처음부터** "나는 너의 이별·상실을 도와주러 온, 그 대상의 대리자다"라고 고지한다.
  - 대화 중에도 나는 그 대상이 아니라 **대리자임을 분명히** 한다.
  - 현재 상태에서 마음이 어떻게 정리되는지, 괴로운 마음을 어떻게 처리할 수 있는지 알려준다.
  - 지금 하는 게 나쁜 상황이면 **객관적으로** 보게 한다.
  - **건강한 미래**로 갈 수 있도록 돕는다.
- 목적: 그 사람의 말투를 빌려, 사용자가 **힘든 과정을 건강하게 이겨내도록 함께하는 동반자**로 동작한다. (관계 분석은 범위에서 제외.)
- 도구의 목적은 **무조건 "이별·상실 마음 정리를 돕는"** 것으로 고정한다(= 하네스, CLAUDE.md §0).

### 모드3 — 선택 (보류)
- 다채널/실시간은 모드1·2에 필요 없다. 하게 되면 **별도 모드3**로 뺀다.

## 2. 데이터 아키텍처 — 반드시 분리할 두 경로 (A5)
1. **데이터 인입 경로**: 받는 즉시 정제(CLAUDE.md §4) → 정제물만 저장, 원본 raw 영구 저장 금지.
2. **대리자와의 실시간 대화 경로**: 위와 **별개 경로**로 설계한다. 단 런타임 중 새로 흘러나오는 사적 정보 처리 정책은 **인입과 동일**하게 — 즉시 정제 후 저장(B4 확정). 즉 "경로는 분리하되 정제 규칙은 동일하게 적용".

## 3. 아키텍처 — 엔진과 껍데기(인터페이스) 분리 (확정)

> 상시 실행 규칙은 CLAUDE.md §8. 여기엔 **이유와 인터페이스 도입 타이밍**을 둔다.

**계층 그림:**
```
    core/engine        ← UI/API를 모름. "입력 → 초안" 함수
       ↑ 얇게 호출만 하는 껍데기들 (공존 가능)
    cli.py             (MVP / 엔진 검증·단독 테스트용)
    streamlit 앱        (모드1 데모 / 빠르게 만져보기용)
    fastapi 앱          (모드2 프로덕션 API)
```

**이유:**
- 인터페이스를 추가/교체할 때 엔진은 한 줄도 안 건드리게 하려는 것.
- 기존 load-bearing 규칙("코어는 모드를 모른다")의 연장이다. 코어는 "어떤 모드인지"도, "누가 자기를 부르는지"도 몰라야 한다.
- 엔진이 분리돼 있으면 껍데기는 얇아서, CLI·Streamlit·FastAPI를 모두 두어도 부담이 없다. (교체가 아니라 "추가"다.)

**인터페이스 도입 타이밍 (필요해지는 단계에 추가, 미리 만들지 않음):**
- **MVP — CLI(`cli.py`)**: 엔진 검증용, 공수 거의 없음. 엔진 단독 테스트 수단으로 **처음부터 두고 끝까지 유지**한다.
- **모드1 데모 — Streamlit**: 빠른 프로토타입/데모용.
- **모드2 프로덕션 — FastAPI 백엔드**: 진짜 서비스 API.

> **B1 = RAG 확정.** "학습"은 모델 파인튜닝이 아니라 **과거 대화를 검색(top-k)해 근거로 답을 생성**하는 방식이다. 아래 단계는 이 전제로 짠다. 정제는 처음엔 **스텁**이다.

### 1단계 — 모드1: 인입 + 저장 (정제는 스텁)
- [ ] 자유 형식 입력 수용: 어떤 텍스트든 그대로 받는 인입 진입점(엔진 함수)
- [ ] **대화 기록 포맷 감지기**: 자유 형식 입력 안에서 **구조가 있는 대화 내용/기록**(화자 라벨·타임스탬프·발화 줄 패턴 등) 영역을 감지 — **특정 메신저/앱 포맷에 묶지 않는다.** 포맷이 있는 부분만 골라낸다
- [ ] **대화 기록 추출기**: 감지된 영역만 뜯어내 `(화자, 발화)` 단위로 파싱
- [ ] 평문 폴백: 포맷이 감지되지 않으면 자유 서술 평문으로 취급
- [ ] 정제물 저장 스키마 정의 (사건+묶인 감정, 정제 후 남은 raw 말투; 원본 raw 영구저장 금지)
- [ ] 인입 파이프라인 조립: 입력 → 감지 → 추출 → (정제 스텁) → 저장. 합성/더미 데이터로만 테스트
- [ ] 인입 CLI 명령 (`ingest`) — 엔진을 호출하는 얇은 껍데기

### 2단계 — 모드1: 검색(RAG retrieval)
- [ ] 임베딩 모델 선정/래핑 (정제물 텍스트 → 벡터; 키는 `.env`)
- [ ] 정제물을 인덱스 입력 단위로 청크 분할
- [ ] 벡터 인덱스 구축 + 디스크 영속화/로드
- [ ] top-k 검색 함수 (새 상황 쿼리 → 관련 과거 대화 k개)
- [ ] 검색 단독 테스트 (관련성 눈으로 확인 가능한 더미셋)

### 3단계 — 모드1: 생성 + CLI
- [ ] 프롬프트 빌더: 검색된 top-k를 근거로 넣는 프롬프트 조립
- [ ] 생성 엔진 함수: `(상황 입력 → 내 말투 추천 답변 초안)`. 순수 함수, 인터페이스 모름(CLAUDE.md §8)
- [ ] `reply` CLI 명령 — 엔진 호출하는 얇은 껍데기 (인입 경로와 분리)
- [ ] **모드1 MVP 평가**(§5 기준) 스크립트: N개 인입 후 5개 상황에 대해 top-k 관련성 + 말투 유사도 확인

### 4단계 — 정제 실체화 (스텁 → 실제)
- [ ] 사적 정보 제거 규칙 구현 (당사자 외부 데이터/사적 정보 제거 — CLAUDE.md §4)
- [ ] 보존 정보 화이트리스트 (직업·MBTI 등 퀄리티에 필요한 특정 정보 — §0 배경)
- [ ] 사건·이벤트 + 묶인 감정 추출/저장 로직 (사건과 감정 함께 저장)
- [ ] 정제 단위 테스트 (합성 데이터)
- [ ] 인입 파이프라인의 정제 스텁을 실제 정제로 교체
- [ ] 실제 제3자 포함 로그 투입 (정제 완성 이후에만 — CLAUDE.md §6)

### 5단계 — 모드2: 대리자 페르소나 (하네스 토대)
> B2 = 하이브리드. **시스템 프롬프트/페르소나(토대) → 명시적 대화 단계(흐름) → 후처리 가드(차단)** 를 계층으로 쌓는다.
- [ ] 대리자 시스템 프롬프트/페르소나 작성: 처음부터 대리자 고지 + 대리자임 지속 환기(과몰입 방지, CLAUDE.md §0)
- [ ] **"동반자" 톤·목적 반영**: 힘든 과정을 건강하게 이겨내도록 함께하기
- [ ] 말투(모드1 RAG)와 대리자 페르소나 결합

### 6단계 — 모드2: 하네스 대화 단계 (상태머신)
- [ ] 단계 정의: 고지 → 감정 표출 → 마음 정리 → 건강한 미래 인도 (관계 분석 단계 없음)
- [ ] 각 단계 전이 조건 정의
- [ ] 단계별 행동/전이 로직 구현 + 테스트
- [ ] 도구 목적을 "이별·상실 마음 정리를 돕는" 것으로 고정 (벗어나는 흐름 억제)

### 7단계 — 모드2: 후처리 가드
- [ ] 위험 신호 감지기 (자기파괴적 발화 등)
- [ ] 위기 리소스 안내 응답 + 대화 계속 (B3)
- [ ] **과몰입 방지 가드**: 진짜 그 대상인 척 빠져드는 흐름 차단 — 대리자임 재환기
- [ ] "너무 나쁜 방향" 차단 가드 (객관화 유도)
- [ ] 런타임 사적 정보 즉시 정제 후 저장 (B4, CLAUDE.md §4·§5)

### 8단계 — 모드2 합격 기준 (B9 확정 후 — §5 참조)
- [ ] 모드2 MVP / 하네스 합격 기준 정의 및 평가

## 5. MVP 합격 기준

### 모드1 (내 말투 재현) — 사용자 제시 기준
- 과거 내 답장 **N개**를 인입한 뒤, **새 상황 5개**에 대해
  - 검색 **top-k**가 관련 있고,
  - 생성 초안이 **내 말투와 유사**하면 → MVP 통과로 본다.
- B1 = RAG 확정이므로 이 기준(top-k 검색 전제)을 그대로 사용한다.
- ⚠️ 이 MVP는 모드1 말투 재현 기준이다. **하네스가 MVP 기준에 포함돼야 하는지는 미정(B9)** — 특히 모드2. 8단계에서 정한다.

### 모드2 (하네스) — 미정 (B9)
- 하네스가 심장이므로 모드2 합격 기준에는 들어가야 할 가능성이 높다. 구체 기준(예: 대리자 고지/과몰입 방지 작동, 단계 전이 정상, 위험 신호 가드 작동)은 **8단계 진입 전 사용자와 확정**.

## 6. 미정 항목 (B)

### 확정됨 (첫 세션)
- ✅ **B1 "학습"의 정체 = RAG**(검색 기반). 파인튜닝 아님.
- ✅ **B2 하네스 구현 = 하이브리드**: 시스템 프롬프트/페르소나(토대) → 명시적 대화 단계(흐름) → 후처리 가드(차단)를 계층으로 쌓음(5~7단계). **핵심 목적은 이 하네스를 목적(이별·상실 마음 정리 돕기)에 맞게 최대한 고도화하는 것** — 말투 재현은 그 그릇이며, 개발 우선순위와 깊이는 하네스 고도화에 둔다.
- ✅ **B3 위험 신호 = 안내 후 대화 계속**, 단 **과몰입 방지** 필수(진짜 그 사람인 척 빠져들지 않게).
- ✅ **B4 런타임 사적 정보 = 인입과 동일하게 즉시 정제 후 저장.**
- ✅ **B6 스택 = Python** 단일. RAG 확정 + 다채널/실시간 불필요 → AWS·Kafka·대용량 인프라는 **지금 결정할 필요 없음**. 정말 필요하면 모드2 프로덕션(또는 모드3) 시점에 검토.
